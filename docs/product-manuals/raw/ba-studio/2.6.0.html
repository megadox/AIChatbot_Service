01. BA Studio 소개

BA-Studio란 RPA (Robotic Process Automation- 아래 RPA 정의 참조) 개발자, 반복 업무의 자동화를 필요로 하는 다양한 기관 및 산업 분야의 현업 담당자 또는 프로젝트 매니저, 미래의 자동화 분야를 이끌어 갈 학생들을 포함한 모든 관심 있는 분들에게 제공되는 업무 자동화 개발 도구이다. 자동화 태스크 구현을 위해 Drag&Drop방식과 Flow Chart형식의 편리한 그래픽 User Interface와 업무 자동화에 필요한 다양한 단위 기능들을 액티비티 형태로 제공하고 있다.

RPA는 인간이 수행하던 규칙 기반의 단순 반복적인 업무를 소프트웨어를 통하여 자동화하여, 저렴한 비용으로 신속 정확하게 업무를 수행할 수 있는 디지털 노동력(Digital Workforce)을 제공하는 로봇의 역할을 하는 소프트웨어 프로그램으로 정의할 수 있으며 아래의 특징을 가지고 있다.

1-1. RPA 정의

RPA는 인간이 수행하던 규칙 기반의 단순 반복적인 업무를 소프트웨어를 통하여 자동화하여, 저렴한 비용으로 신속 정확하게 업무를 수행할 수 있는 디지털 노동력(Digital Workforce)을 제공하는 로봇의 역할을 하는 소프트웨어 프로그램으로 정의할 수 있으며 아래의 특징을 가지고 있다.

1-2. BA Studio의 역할

BA Studio와 실행환경을 포함한 동작시퀀스 연결을 아래의 그림에 나타내었다.

1) BA Studio의 역할

업무자동화 개발 툴인 BA-Studio는 자동화를 위한 다양한 단위 기능을 수행하는 액티비티와 사용하기 편리한 개발환경을 제공하여, 개발자 또는 현업 담당자들로 하여금 쉽게 자동화 업무 태스크를 구현하고 실행할 수 있게 한다.

BA-Studio에 의해 만들어지는 자동화 태스크 구현의 결과물은 아래의 세 가지 파일 형식을 갖는다.

파일형식 설명
fp파일 (*.fp) BA-Studio에서 업무 자동화를 위한 태스크를 생성하여 구현한 실행파일로써, 파일 확장자는 fp이다.
fpp파일(*.fpp) BA-Studio에서 업무 자동화를 위한 프로젝트를 생성하여 구현한 실행파일로써, 파일 확장자는 fpp이다.
bll파일(*.bll) 프로젝트에서 사용하기 위해 BA-Studio에서 생성하는 라이브러리 파일이다. 파일 확장자는 bll이다.

BA-Studio를 통해 구현된 fp파일 또는 fpp파일은 BA-Studio안에서 독립적으로 동작할 수 있고, 필요한 리소스와 함께 묶여 패키지 파일인 fpk파일, fpx파일로 변환되어 BA-Assist 또는 BA-Server로의 입력파일로 사용될 수도 있다. 라이브러리로 사용할 수 있는 bll파일 역시 BA-Studio에서 생성되고, 공통으로 사용하기 위해 BA-Server에 업로드 한다. 이때 패키지 파일인 fpl 파일로 변환된다. 패키지 파일의 생성과 사용 단계는 Appendix A. BA-Studio의 인터페이스에 나타내었다.

2) BA Server의 역할

BA-Server는 서버사용자(또는 관리자)로 하여금 BA-Studio에 의해 구현된 업무자동화 태스크들을 스케줄러의 설정에 따라 Bot들에게 배정하여 수행하게 한다. 또한 태스크들의 실행과정을 모니터링하고 실행 조건을 조정하고 관리하는 역할을 담당한다.

3) BA Worker의 역할

BA-Server에 접속하여 BA-Sever로부터 지시 받은 자동화 업무를 일시적으로 또는 반복적으로 정확히 수행하는 기능을 담당한다.

위에서 언급한 BA-Server와 BA-Worker외에 독립 PC에서 스케줄기반의 태스크수행을 담당하는 BA-Assist까지를 포함한 BA-Studio의 주변 인터페이스에 관해서 #Appendix A. BA-Studio의 인터페이스에 표시된 관계도를 참조한다.

1-3. BA Studio의 특징

BA-Studio는 Drag&Drop 방식의 손쉬운 Work Flow작성 기능과 Code기반의 프로그래밍을 더하여 확장성을 고려한 개발 환경을 제공한다.

내용 확인이 용이한 화면 확대/축소 기능, Flowchart 형식의 개발 UI를 제공하여 소스 분석을 용이하게 하며, 이미지 인식기술, 좌표 인식기술, Object인식기술 기반의 자동화 방식을 지원하고 있다.

BA-Studio는 Engine Level에서의 구현 기능 확장을 위한 .Net Framework 기반 구성과 Code Level에서의 구현 기능의 확장을 위한 Python 스크립트 방식을 지원하고 있으며, 자동화 실행 중 문제가 발생하면 바로 확인 및 수정/재실행을 할 수 있는 Debugging 기능을 제공하고 있다.

  • ㆍDrag&Drop 방식의 Work Flow 작성 방식 제공
  • ㆍ이미지, 좌표, Object인식기술 기반의 다양한 프로세스 구현 기능 제공
  • ㆍ프로세스 자동화 결과물인 태스크의 실행 시, 관리자도 Work Flow의 진행상태를 알 수 있도록 하기 위해 현재 진행 중인 상태를 시각적으로 Focus하는 User Interface 제공
  • ㆍ.Net Framework 와 Python Program 을 통한 기능 확장성 제공
  • ㆍWork Flow 수정과 재구현를 위한 Debugging 기능 제공

1) 태스크 실행의 예외 사항

태스크가 한 번도 저장되지 않은 상태에서 실행 버튼 을 클릭할 경우 저장화면으로 이동하여 저장한 후
실행을 수행한다. 단, 다이어그램이 없거나 Start만 있을 경우에 실행 버튼을 누르면 태스크를 실행할 수 없음을
나타내는 알림메시지가 표시된다.

2) 다이어그램(액티비티) 자동 연결 방식

액티비티 다이어그램에서 새로운 액티비티를 추가, 이동, 삭제 및 붙여넣기할 때 자동으로 연결을 설정하고 위치를 정렬하는 기능을 제공합니다. 본 메뉴얼에서는 해당 기능의 동작 방식을 상세히 설명합니다.

기능 설명

1. 새로운 액티비티 추가

- 사용자가 새로운 액티비티를 드래그하여 추가하면, 마우스 위치를 기준으로 액티비티가 삽입됩니다.

- 현재 마우스 위치에 겹치는 액티비티가 있는 경우:
   * 겹치는 액티비티 하위에 신규 액티비티 추가 및 자동 연결.
   * 상하좌우 간격을 고려하여 자동 정렬 수행.


- 겹치는 액티비티가 없는 경우:
   * 최하단 액티비티 하위에 신규 액티비티 추가 및 자동 연결.
   * 전체 다이어그램을 정렬하여 일관된 배치를 유지.


2. 기존 액티비티 이동

- 사용자가 기존 액티비티를 드래그하여 이동하면, 마우스 위치를 기준으로 새로운 위치로 이동됩니다.

- 현재 마우스 위치에 겹치는 액티비티가 있는 경우:
   * 기존 위치에서 액티비티를 제거하고, 겹치는 액티비티 하위에 이동 및 자동 연결.
   * 상하좌우 간격을 유지하며 자동 정렬 수행.


- 겹치는 액티비티가 없는 경우:
   * 이동이 취소되며, 원래 위치로 복귀.


3. 액티비티 삭제

- 사용자가 액티비티를 삭제하면, 해당 위치를 기준으로 위/아래 액티비티가 자동으로 연결됩니다.

- 삭제 방식:
   * Delete 키 입력: 선택된 액티비티 삭제 후 자동 연결 및 정렬.
   * Ctrl + X (잘라내기): 선택된 액티비티를 클립보드에 복사 후 삭제, 위/아래 액티비티 자동 연결 및 정렬 수행.


4. 액티비티 붙여넣기

- 사용자가 복사(Ctrl + C) 또는 잘라내기(Ctrl + X)한 액티비티를 붙여넣기(Ctrl + V)할 때 다음 규칙이 적용됩니다.

- 선택된 액티비티가 있는 경우:
   * 선택된 액티비티 하위에 붙여넣기 수행 및 자동 연결.


- 선택된 액티비티가 없는 경우:
   * 최하단 액티비티 하위에 붙여넣기 수행 및 자동 연결.


주의사항

- 액티비티 추가, 이동, 삭제, 붙여넣기 시 다이어그램의 논리적 구조를 유지하기 위해 자동 정렬됩니다.

- 액티비티 간 연결이 끊어지지 않도록 자동 연결이 수행됩니다.

- 사용자가 작업을 수행할 때마다 전체적인 다이어그램의 균형을 유지하는 방식으로 자동 정렬이 적용됩니다.

Decision 참(True) 또는 거짓(False) 여부에 따라 두 가지 경로로 작업 흐름을 분기시키는 액티비티.
DoWhile 작업을 최소 한 번 실행한 후, 조건이 거짓일 때까지 계속 반복하는 액티비티.
ForEach 컬렉션(리스트, 딕셔너리, 세트 등)의 각 항목에 대해 반복적으로 작업을 수행하는 액티비티.
While 조건이 참(true)인 동안 반복적으로 특정 작업을 수행하는 액티비티.
MultiThread 동시 병렬 처리를 지원하는 액티비티.
SwitchCase 여러 조건을 검사하여, 조건 값과 일치하는 케이스를 실행하는 액티비티.
TryExcept 작업 중 발생할 수 있는 오류를 처리할 수 있도록 예외를 관리하는 액티비티.

03. 태스크 / 프로젝트 / 라이브러리

3-1. 용어의 정의

아래의 용어의 정리는 단순히 BA-Studio를 활용하여 프로젝트를 구현할 때 도움이 되도록 하기 위함이다.

1) Task (태스크)
프로젝트를 구성하는 자동화된 업무 또는 자동화될 업무를 태스크라고 정의한다.
2) Process (프로세스)
태스크의 진행 시퀀스를 프로세스라고 한다.
3) Work Flow (워크플로우)
BA-Studio에서는 프로젝트를 구성하는 태스크 플로우와 태스크를 구성하는 액티비티 플로우를 합쳐서 워크 플로우로 정의할 수 있으며, 사용자 입장에서는 업무자동화를 위한 일을 흐름을 나열할 때 사용할 수 있다.
4) Activity (액티비티)
태스크 내의 일의 단위 수행업무를 액티비티라고 하며, 프로그램 관점으로 보면 하나의 기능을 수행하는 Method 또는 Function에 해당한다. BA-Studio에서 제공되는 Activities는 용도별로 여러 그룹으로 나뉘어져 있다.
5) Property (속성)
액티비티 동작에 필요한 정보를 담고 있는 속성값 또는 전달 파라미터들을 의미한다.
6) Variable (변수)
프로젝트 내에서 태스크 간에 공유하며 읽기/쓰기 가능한 Global 변수와 특정 태스크 내에서만 읽기/쓰기 가능한 Local 변수로 나눌 수 있다.
태스크간 자원공유를 원할하게 하기 위해 paramter 역할을 하는 Input 변수, 결과를 반환할 수 있는 Output 변수가 추가되었다.
Input / Output 변수는 프로젝트에서 Predefined-Task, Library 사용시 유용하게 자원을 공유 할 수 있다.
7) Library (라이브러리)
특정한 부분 기능을 수행하도록 제작된 Task 파일들의 모음이다. BA-Studio에서 작성하고, BA-Server를 통해 공유되며, Project에서 사용할 수 있다.

3-2. 태스크 구현하기

3-2-1. 태스크 구현의 주요방법

태스크를 구현하는 주요 방법으로 ① Image 인식기반 구현방법과 ② Object 인식기반 구현방법이 있다. 여기에 ③ 좌표 인식기반 구현방법을 포함하여 세가지 방법을 혼용하여 태스크를 구현할 수 있다.

① Image 인식기반 구현방법

RPA가 자동화의 대상이 되는 애플리케이션이 무엇인지에 상관 없이 사용자 동작을 모방하여 따라할 때 사용되는 방법으로, 마우스 클릭 또는 키 입력의 대상이 되는 부분의 이미지를 캡쳐하여 보관했다가 RPA가 사용자의 모방 동작을 수행할 때 이미지 매칭을 통하여 마우스 클릭이나 키 입력의 위치를 판단하는데 사용하는 방법이다.

② Object 인식기반의 구현방법

브라우저 또는 윈도우 애플리케이션에서 제공하는 Software Interface를 활용하여 마우스 클릭 또는 키 입력의 대상을 오브젝트 단위로 기억하였다가 RPA의 모방 동작 시에 활용하는 방법이다.

③ 위치기반의 구현방법

윈도우 화면의 특정 좌표(x, y) 위치에 마우스를 클릭하는 방법을 제공함으로써 커서의 포거스 위치를 옮기게 할 수 있다. 이 때 마우스 클릭할 좌표값은 대상 윈도우화면을 복사한 후 그림판에 붙여넣기 하여 알아낼 수 있다. 이 위치 기반의 작성법은 개발 PC의 환경 변화에 영향을 받기 때문에 유의하여 사용해야 한다.

3-2-2. 변수의 사용

일반적인 Programming언어와 같이 BA-Studio도 지역변수와 전역변수의 사용법을 제공하고 있다.
지역변수란 한 태스크 내에서만 영향력이 있고 선언된 아래의 부분에서만 값의 읽기/쓰기가 유효한 변수이고, 전역변수는 Task간에 상호 참조/변경이 가능한 변수이다.
Prefined-Task, Library 에서 원활하게 자원을 공유 할 수 있도록 Parameter 역할을 하는 Input 변수, 결과값을 공유할 수 있는 Output 변수가 있다.


☞ Task 내에서 사용하는 지역변수, 전역변수, Input / Output 변수는 모두 Variables 창에서 추가하여 사용해야 한다.
Variables 창에서 선언된 변수는 Flow창, Code 창, Code Editor, Text Editor에서 사용할 수 있다.
Code창에서 선언된 변수는 관리되지 않는 변수이므로 Flow창이나 다른 Editor에서 사용 할 수 없다.


▶ Variables 창 기본 사용
  • 선언 : Variables창에서 마우스 우클릭 후 추가
  • 새로운 Row에서 scope, type, key, value, description등을 입력한다. scope, type, key는 필수 값이다.
  • 변경사항 저장 버튼을 클릭하여 저장해야 한다.
  • 삭제 : 마우스 우클릭 후 삭제
▶ 지역변수의 선언과 사용법
  • scope : Local 선택
  • Code 창 , Code Editor, Text Editor 자동완성 툴팁에 Local Variables이 표시된다.
  • 타입에 맞게 파이썬 문법에 따라 사용한다.
  • 지역변수이므로 해당 Task에서만 사용할 수 있다.
▶ 전역변수의 선언과 사용법
  • scope : Global 선택
  • Code 창 , Code Editor, Text Editor에서 글로벌을 나타내는 g.을 입력하면 전역변수가 표시된다.
  • 타입에 맞게 파이썬 문법에 따라 사용한다.
  • 전역변수이므로 프로젝트 전체에서 사용할 수 있다.
▶ Input / Output 변수의 선언과 사용법

☞ Flow 창에서 빈공간을 클릭하면 Properties는 파일 Propertie를 보여준다.
입력/출력 변수관리창에서 추가된 Input/Output 변수는 Variables창에도 추가된다.


  • ● 파일 Properties에서 Task의 Input/Output을 추가할 수 있다.
  • ● 또는 Variables 창에서 scope를 Input/Output으로 지정하여 선언 할 수 있다.
  • ● Code 창 , Code Editor, Text Editor 자동완성 툴팁에 Input Variables / Output Variables이 표시된다.
  • ● 타입에 맞게 파이썬 문법에 따라 사용한다.
  • ● Predefined-Task로 사용할 때 파라미터 역할을 한다.

- Task1에서 Tas2를 Predefined-Task로 사용한다. ( Project 트리에서 Task2를 Drag&Drop 해서 원하는 위치에 놓는다.)
- 사용된 Task2를 클릭하여 속성을 확인한다.
- 속성창에서 Input/Output 클릭하여 입력출력 변수관리 창을 팝업한다. Task2에서 정의했던 Input/Output 변수를 확인 할 수 있다. Predefined-Task에서는 추가및 삭제는 할 수 없고, 변수의 Value만 할당 할 수 있다.

3-2-3. 태스크 흐름제어

태스크에서 구현되는 Work Flow의 흐름제어를 담당하는 기본적인 액티비티는 (Decision), (ForEach), (DoWhile), (While), (MultiThread), (SwitchCase), (TryExcept) 총 7가지이다. 아래의 예제를 통하여 각 플로우의 기본적인 동작과 더불어 변수의 사용 예도 함께 알아보기로 한다.

▶ Decision (판단문, If~Else~분기문)

▶ ForEach (반복문, ForEach Loop)

Loop(순환문)은 특정 액티비티의 수행을 반복하도록 하는 역할을 한다. BA-Studio는 일반적인 프로그래밍 언어의 ForEach 문을 약간 변형하여 사용하고 있다. ForEach Loop문은 Enhanced For Loop 이라고도 불리우며 List나 Tuple 등의 컬렉션 데이터에 들어 있는 모든 항목에 대하여 반복구문이 수행되도록 구현되어 있다.

  • collection: 루프를 통해 반복하게 될 컬렉션 자료의 레퍼런스 값이다. 레퍼런스란 객체지향형 프로그램에서 오브젝트의 위치를 저장하고 있는 변수를 말하며 BA-Studio에서는 간단히 컬렉션 데이터를 저장하는 저장 변수라고 지칭한다.
  • Index(): 0에서 시작하여 ForEach() Loop 이 반복될 때마다 1씩 증가한 값을 반환한다.
  • Item(): ForEach Loop이 반복될 때마다 컬렉션 데이터의 항목을 하나씩 반환한다. 리턴 항목은 컬렉션 내부의 각기 다른 원소(=element) 하나 하나를 의미한다.
  • 반복횟수 N: collection의 원소(=element)의 개수가 N개 일 때, Loop은 N번 반복한다.
  • Body: Loop안에서 반복하여 수행하는 Activity 또는 Predefined Process들의 시퀀스(=나열)이다.

아래에 BA-Studio에서 사용하는 Syntax를 나타내었고 예제를 통하여 사용법을 구체적으로 알아보도록 하자.

▶ Multi Thread (다중 처리, Multi Threading)

스레드란 프로세스 내에서 작업을 수행하는 주체를 의미한다. 기본적으로 프로젝트 또는 태스크 실행 시, 하나의 프로세스에서 하나의 메인 스레드가 실행된다.
BA-Studio에서는 Multi Thread 액티비티를 사용하여 사용자가 여러 작업을 동시에 처리하여 작업 속도를 획기적으로 줄일 수 있도록 지원한다.

아래 예제를 통해 사용법을 구체적으로 알아보자.

기본 사용법은 다음과 같다.
1. MultiThread 액티비티를 에디터에 끌어다 놓는다.
2. "Add Thread"를 클릭해서 실행할 Thread를 추가한다.
3. 각각 Main Thread, Thread 1, Thread 2... 에 실행 할 액티비티를 연결한다. (주로 SubTask를 연결한다. )

주요 Property 목록은 다음과 같다.

  • range: range 속성은 연결된 각각의 스레드에 리스트 객체의 순서에 맞는 요소값을 할당하여 요소의 길이만큼 동시 반복 실행하고자 할 때 사용한다.

    1) 데이터는 2차원 리스트를 할당한다.
    2) 2차원 리스트의 길이는 Thread 수와 같아야 한다.
    3) range 속성에 값이 없을 경우, Thread들을 한 번씩 실행한다.

  • breakOnError: 다중 처리 실행 중 스레드에서 에러 발생 즉시 실행중인 모든 스레드를 종료시킬지, 다른 스레드들의 작업이 완료될 때까지 기다릴 지 여부를 선택한다. (True/False)

3-4. Debugging 사용하기

BA-Studio에서 중단점 설정, 단계별 실행, 실행중 변수 확인 등 디버깅 기능을 제공하여 태스크 및 프로젝트를 구현할 때 도움이 되도록 하기 위함이다.

BA-Studio 에서 제공하는 Debug 기능에는 중단점 설정, 여기서 부터 실행, 계속 실행, 단계별 실행, 핫 다시 로드, Debug 명령줄, 변수 확인 및 수정 등 기능을 제공하여 보다 쉬운 프로젝트 또는 태스크 개발을 지원한다.


디버깅은 태스크 디버깅과 프로젝트 디버깅으로 구분하여 동작한다.

* Debug Task : 현재 활성화된 태스크를 디버그 실행한다.
* Debug Project : 현재 프로젝트를 디버그 실행한다.


멀티스레드 내부의 태스크에서는 디버깅을 지원하지 않습니다.

1) 중단점 설정

중단점을 설정하는 방식은 2가지 방법이 있으며 아래와 같다.


1. 액티비티 우클릭 컨텍스트 메뉴 사용

중단점을 걸고자 하는 액티비티의 제목에 마우스 우클릭을 하면 컨텍스트 메뉴가 나타나며, 여기서 Debug > Set BreakPoint를 선택하면 중단점이 설정된다.

* 중단점 설정 단축키 : Ctrl + G


2. 액티비티 프로퍼티 BreakPoint 사용

중단점을 걸고자 하는 액티비티의 프로퍼티에 BreakPoint를 True로 변경하면 중단점이 설정된다.

2) 여기서 부터 실행

특정 액티비티부터 디버그를 실행하는 방법은 아래와 같다.


여기서부터 디버그를 실행하고자 하는 액티비티의 제목에 마우스 우클릭을 하면 컨텍스트 메뉴가 나타나며, 여기서 Debug > Debug From Here를 선택하면 앞의 액티비티는 생략되고 디버그를 실행한다.

3) 계속 실행

디버그 실행 이후 중단점에서 프로세스가 멈췄을 경우 디버그 계속 버튼을 누를 경우 다음 중단점이 있다면 다음 중단점까지 없다면 프로세스 마지막까지 계속 실행된다.


4) 단계별 실행

디버그 실행 이후 중단점에서 프로세스가 멈췄을 경우 디버그 한 단계 실행 버튼을 누를 경우 중단점 다음 액티비티를 실행한다.


5) 핫 다시 로드

디버그 실행 이후 중단점에서 프로세스가 멈추고 태스크 파일이 수정되었을 경우 디버그 핫 다시 로드 버튼을 누를 경우 태스크 파일을 다시 로드하여 수정된 태스크 파일의 내용으로 디버그를 계속 실행한다.


6) Debug Console

디버그 실행 중 디버그 로그를 표시하고 명령어를 입력할 수 있는 Debug Console창을 제공한다.
디버그 콘솔창은 디버그 실행시 디버그 콘솔창을 통해 디버그 실행 결과를 확인할 수 있으며 하단에 명령창을 통해 디버그 실행을 제어할 수 있다.


위쪽 화살표 및 아래쪽 화살표 키를 사용하여 명령 기록을 스크롤할 수 있습니다.

7) Debug Frames

디버그 실행 중 현재 프레임을 표시하는 Debug Frames창을 제공한다.
디버그를 실행하면 현재 프로세스가 실행되는 프레임을 프레임 스택에 쌓고 디버그 프레임창을 통해 현재 실행 Frame을 확인할 수 있다. 프레임은 현재 프로세스가 실행 중인 태스크의 이름과 태스크 파일의 위치를 표시한다.


8) Debug Variables

디버그 실행 중 변수의 값을 확인하고 수정할 수 있는 Debug Variables창을 제공한다.
변수 목록은 Global 변수와 Local 변수가 Group: G와 Group: Variables에 구분되어 담겨있으며 변수의 값을 확인하고 수정할 수 있다.


* Value 부분에 값을 수정하여 변수의 값을 변경할 수 있으며 해당 변경은 현재 디버그 실행 중에만 변경 적용된다.
* 변수의 값을 변경할 경우 python에서 사용하는 문법을 이용하여 변경할 수 있다.(ex.문자열은 python에서 문자열을 나타내는 "(따옴표) 또는 '(작은따옴표)를 이용하여 변경해야 한다.)

9) Debug BreakPoints

디버그 실행 중 중단점을 확인할 수 있는 Debug BreakPoints창을 제공한다.
BreakPoints 창은 중단점이 설정된 액티비티의 목록을 표시하며 해당 목록을 클릭하여 중단점이 설정된 액티비티로 바로 가기 할 수 있다.

약어 설명
M/O Mandatory or Optional? 필드 값의 입력이 필수 사항인지 선택 사항인지를 나타낸다.

C Changeable or not? 필드 값이 변경 가능한지 아닌지를 표시한다.
Y: Yes, N: No

4-2-3. EXCEL Group Activities

Excel Group Activities의 동작을 알아보기 전에 최상위 개념인 Excel Instance란 무엇인지에 관해 먼저 알아보기로 한다. Excel 은 Instance 에 종속적인 기능들을 제공하여 하나의 Instance 안에 있는 파일들끼리 정보를 교환할 수 있도록 구현되어 있다. Excel 2013 버전 이후에 Instance간 모니터링 기능 등 새로운 인터페이스 기능들이 추가되고 있는데, 여기서는 One Instance 와 Multi-Files 관계를 가지고 있다는 것만 이해하고 시작하자.

Instance 관련한 엑셀 일반동작 참고사항

  1. File-New Workbook 또는 Ctrl+N 에 의해 생성된 모든 파일들은 같은 Instance 아래 있어야 한다.
  2. Ctrl+F6 누르면, 같은 Instance안에 열려 있는 다음 파일을 먼저 Open하게 된다.
  3. Task Manager를 열었을 때, 하나의 Excel.exe 만이 보인다면, 거기엔 오직 하나의 Instance 가 동작하고 있음을 의미한다.

노트1 EXCEL Activities Group에서 CreateInstance가 먼저 호출되고 이어 CreateWorkbook이 호출되어 새로운 Workbook이 만들어지면 Excel Instance 레퍼런스인 excel_0 이 Target필드에 생성되고, 그렇지 않고 CreateInstance 없이 바로 CreateWorkbook이 호출되어 Workbook이 만들어지면 Default() 값이 Target필드에 생성된다.

4-2-6. EMAIL Group Activities

EMAIL Group Activities는 웹브라우저를 접속하지 않고 간편하게 이메일을 전송하기 위한 유틸리티 기능을 제공하는 Activities의 그룹으로 SMTP 와 Outlook 두 가지를 지원한다. 두 가지 모두 단 하나의 액티비티만을 이용하여 이메일을 전송할 수 있는 기능들이다. 다만, SMTP는 SMTP 서버 접속을 위한 Properties설정이 필요하나, Outlook은 이미 PC에 설치되었다면 서버접속을 위한 설정 과정은 필요하지 않다.

1) Send Mail(SMTP): SMTP 설정을 통하여 이메일을 전송한다.

이 액티비티를 이용하여 이메일을 전송하기 위한 Properties 설정은 크게 ① 발신/수신자 이메일 ID, ② 메일 제목/내용/첨부파일, ③ SMTP서버접속정보, ④ SMTP사용자(클라이언트) ID/PW 정보 등으로 아래의 설명을을 참조한다.
